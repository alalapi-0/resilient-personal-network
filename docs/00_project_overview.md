# 00 项目总览（Project Overview）

## 1. 这个项目是什么
`resilient-personal-network` 是一个面向个人用户的网络通道管理工程仓库。  
它用于统一管理以下内容：
- 多个 VPS 节点的规划与记录；
- 服务端/客户端配置模板；
- 初始化与巡检脚本；
- 安全规范与故障处理文档。

简单理解：这是你的“网络通道运维中台仓库”，用于长期维护，不是一次性脚本集合。

## 2. 这个项目不是什么
本项目（尤其 Round 0）不是：
- 一键翻墙工具；
- 真实代理服务安装脚本；
- 可直接对外分发的订阅链接仓库；
- 包含真实密钥和真实服务器地址的私密资料库。

## 3. 为什么要做成工程化项目
如果只靠临时笔记和零散命令，常见问题包括：
1. 配置散落，过几周无法追溯来源；
2. 节点变更后，客户端配置不同步；
3. 故障发生时，缺乏标准排查路径；
4. 备份不规范，恢复困难。

工程化仓库可以把“脚本 + 文档 + 模板 +流程”放在一起，形成可持续迭代的工作方式。

## 4. 为什么不能只依赖单一节点
单一节点存在天然风险：
- 节点宕机：所有链路立即不可用；
- 网络波动：体验明显下降；
- IP 风险：单点被限制后没有替代路径。

因此需要“主节点 + 备用节点”的设计，逐步形成可切换、可恢复、可观测的韧性网络。

## 5. 为什么要做文档和脚本
- **文档**：保证未来的你（或协作者）知道“为什么这样设计”。
- **脚本**：把重复动作标准化，减少手工失误。
- **二者结合**：让系统具备可维护性与可交接性。

## 6. 后续会如何扩展
后续轮次将按“先规范、再自动化、最后运维闭环”的思路推进：
1. 先定义节点信息、模板格式和命名规则；
2. 再实现本地 CLI 生成配置；
3. 然后补齐备份、巡检、故障排查能力；
4. 最终形成多节点故障切换与演练机制。
