# 托管订阅服务策略

## 1. 背景

在“自建 VPS 节点 + 自己维护配置”的路线下，用户可以获得较高的控制权，但也要承担持续运维成本。对个人用户而言，日常可用性往往比“纯自建”更重要。

引入第三方托管订阅服务（常被称为“机场订阅”）的主要原因包括：

- 自建节点可控，但维护成本高（部署、升级、排障、替换 IP 都需要自己完成）。
- 单 VPS 节点抗风险能力弱，一旦故障或不可用，整体链路会中断。
- 托管订阅服务通常提供多个节点，可在故障时切换。
- 对个人用户来说，稳定访问生产力服务通常优先级更高。
- 本项目目标不是执着于某一种技术路线，而是建立稳定、可维护、可切换的个人网络通道体系。

## 2. 三种路线定义

### 2.1 自建路线

**定义：**
用户自己购买 VPS，自己部署服务端，自己生成客户端配置。

**优点：**
- 可控性高。
- 有利于学习底层原理。
- 可与个人自动化系统、远程主机、Agent 项目集成。
- 不依赖第三方机场服务商。

**缺点：**
- 运维压力大。
- IP 被封或协议被识别后需要自己处理。
- 单节点稳定性有限。
- 对新手来说排障成本高。

### 2.2 托管订阅路线

**定义：**
用户购买第三方服务商提供的订阅链接，在 Shadowrocket、sing-box、Clash Meta、Mihomo 等客户端中导入。

**优点：**
- 上手快。
- 通常自带多节点。
- 节点失效时可切换。
- 对手机用户友好。
- 适合快速恢复生产力。

**缺点：**
- 依赖服务商。
- 订阅链接属于敏感信息。
- 服务商跑路、停服、被封时用户较被动。
- 节点质量不透明。
- 隐私信任成本较高。

### 2.3 混合路线

**定义：**
托管订阅作为主力通道，自建节点作为备用通道或学习通道。

这是本项目推荐的现实路线。

**建议结构：**
- 日常使用：第三方机场订阅。
- 备用：自建 VLESS + REALITY 节点。
- 维护：WireGuard / SSH 作为服务器管理或备用链路。
- 本地管理：使用本项目记录、备份、评估、归档所有配置和文档。

## 3. Shadowrocket + 订阅机场的角色

Shadowrocket 是 iOS 上常见的代理客户端。

需要明确：
- Shadowrocket 本身不是机场，也不是服务商。
- 它负责导入订阅、选择节点、执行分流规则。
- 真正提供节点的是订阅服务商。
- 订阅 URL 不能公开。
- Shadowrocket 配置截图可能泄露节点信息。

## 4. 为什么机场订阅通常比单 VPS 更稳定

常见原因包括：
- 服务商通常提供**节点池**，不是单点。
- 节点覆盖**多地区**，可以按网络状况切换。
- 客户端支持**自动或手动切换**。
- 服务商集中维护协议与 IP，用户无需逐个处理故障。
- 单个节点失效不等于整体服务失效。

同时要保持预期管理：
- 稳定不是永久。
- 机场也可能跑路。
- 机场也可能被封。
- 必须准备备用方案。

## 5. 自建与托管路线对比表

| 维度 | 自建单节点 | 自建多节点 | 第三方机场订阅 | 混合模式 |
|---|---|---|---|---|
| 上手难度 | 高 | 很高 | 低 | 中 |
| 稳定性 | 中-低（单点风险） | 中-高（取决于运维能力） | 中-高（取决于服务商质量） | 高（主备分离） |
| 可控性 | 很高 | 很高 | 低-中 | 高 |
| 隐私信任成本 | 低（主要信任自己） | 低（主要信任自己） | 高（需信任服务商） | 中 |
| 成本 | 低-中 | 中-高 | 中 | 中-高 |
| 维护压力 | 高 | 很高 | 低 | 中 |
| 多设备支持 | 需自行适配 | 需自行适配 | 通常较好 | 较好 |
| 自动切换能力 | 弱（需自行实现） | 中（可自建策略） | 中-高（客户端 + 节点池） | 高 |
| 学习价值 | 很高 | 很高 | 中 | 很高 |
| 长期可持续性 | 中（依赖个人精力） | 中-高（需系统化） | 中（依赖供应商） | 高（风险分散） |
| 适合场景 | 技术学习、可控实验 | 有运维经验的长期自管 | 快速可用、移动端优先 | 生产力 + 学习并重 |

## 6. 本项目推荐策略

**短期：**
优先使用 Shadowrocket + 可靠订阅服务，先恢复 ChatGPT / Claude / YouTube / GitHub 等生产力访问。

**中期：**
继续建设自建 VLESS + REALITY 节点，作为备用与学习路线。

**长期：**
将机场订阅、自建节点、客户端配置、故障记录都纳入 `resilient-personal-network` 项目统一管理。

## 7. 托管订阅服务的选择标准

> 不推荐具体服务商，仅给出判断标准。

- 是否支持常见客户端。
- 是否提供标准订阅链接。
- 是否支持多个地区节点。
- 是否有备用入口。
- 是否支持流媒体 / AI 服务访问。
- 是否有合理的退款或试用机制。
- 是否频繁更换域名。
- 是否有公开透明的状态页或通知渠道。
- 是否价格过低到不合理。
- 是否强制要求下载来历不明的专属客户端。
- 是否可以仅用通用客户端导入订阅。

## 9. 敏感信息处理规则

- 不要把订阅 URL 提交到 GitHub。
- 不要把真实节点地址写进公开文档。
- 不要上传含节点详情的 Shadowrocket 截图。
- 不要公开二维码。
- 不要公开 UUID、密码、private key。
- 如必须记录，仅使用 `provider-a` 这类代号。
- 真正敏感内容放在本地 `.env` 或加密笔记中。
